Reading a User Story

01 — What Is a User Story?

A user story describes a feature from the perspective of the person using it. At Supply Bistro we write them in Gherkin format — a structured language with keywords like Feature, Scenario, Given, When, and Then. Every word in these keywords has a job.

FEATURE SPEC — INLINE CART MANAGEMENT SPRINT 41
Feature What capability are we building? 
Feature: Inline Cart Management
As a Who is the user? 
  As a restaurant buyer
I want to What action do they want? 
  I want to manage my cart without leaving the order page
So that Why does it matter? Business value. 
  So that I can quickly adjust quantities and remove items
Background Shared setup for ALL scenarios below 
  Background:
    Given I am on the mobile order creation page
    And I have added "Tomatoes (Case) × 2" to my cart
Scenario One specific test case 
  Scenario: Increase quantity
    Given my cart contains "Tomatoes (Case) × 2"
    When I tap the [+] button next to Tomatoes
    Then the quantity should increase to 3
    And the line total should update to $73.50
    And the cart header should show "Cart (1 item · $73.50)"
    And the cart should auto-save to localStorage
02 — What Each Keyword Means
Keyword Your Job as a Tester Common Mistake
Feature Understand the feature name — this is how you find the test in the QA list in ClickUp. Skipping straight to scenarios without reading the feature title and user goal.
As a [role] Know which account type to log in as when testing. Buyer ≠ Seller ≠ Admin. Testing as the wrong user type and getting false failures or missing the right flow.
I want to This is the core action. If the feature works, this should be possible after your test. Ignoring this — it tells you if the test passed at a high level.
So that The business value. If something works but violates this goal, it's still a bug. Not considering whether the implementation actually achieves this.
Background Run this setup BEFORE every scenario in the feature. It's not optional. Thinking the Background is just for one scenario. It applies to ALL of them.
Given Set up your preconditions. Make the app be in this state before you do anything. Skipping setup and testing from a "clean" state that doesn't match the scenario.
When The exact action you perform. Tap, type, click — be precise. Doing extra steps not in the spec (like refreshing the page mid-test).
Then The expected result. This is what you verify. Pass = matches. Fail = doesn't match. Checking only the first Then and missing the And lines below it.
And Continues the previous keyword. And after Then is another thing to verify. Treating And as optional. If it's listed, it must be checked.
💡 Key Rule

Every Then and And after a Then is a separate assertion you must verify. If a scenario has 4 And lines after the Then, you have 5 things to check, not 1.

04 — Where Does This Live in ClickUp?
⚠️ IMPORTANT

The Feature spec (the Gherkin text above) lives in the Product Backlog or sprint task description. The individual test cases live in Engineering → QA → Test Scenarios & Cases. These are two different places.

Where in ClickUpWhat it ContainsWho Updates It
Engineering → Backlog → Product Backlog The user story task. Contains the full Feature/Scenario spec in its description. Product / Design adds stories. Dev picks them up.
Engineering → Platform Development Sprints → Dev Sprint 41 Implementation tasks assigned to developers this sprint. Developers own these.
Engineering → QA → Test Scenarios & Cases One task per scenario (TC-CART-01 through TC-CART-06). Status: ready for testing. QA writes them. Testers execute them.
Engineering → QA → Test Runs A run record per sprint build. Links to the scenarios tested and their pass/fail status. QA creates the run. Testers update it during execution.
Engineering → Backlog → Bug Tracking Failures from test runs become bug tasks here. Linked back to the failing test. Tester creates the bug. Dev fixes it.

02 — Field-by-Field Guide

Title

REQUIRED

Format: Functional Area | Brief description. Keep it under 100 characters. The title must tell anyone — at a glance — what area is affected and what is wrong.

❌ Never: "cart bug", "fix this", "broken", "test 2".

✅ Good: Mobile | Cart [−] Button Allows Qty Below 1 — area is clear, problem is clear, under 100 chars.

Status

REQUIRED

Must always reflect reality right now. An outdated status misdirects the whole team.

❌ Biggest QA sin: leaving a task in Ready for QA while you are actively testing it. Move it to Testing & Validation the moment you start.

✅ See the Status Cheat Sheet for every valid transition.

Assignee

REQUIRED

When you are testing a task, you are the Assignee. If you open a task and your name is not on it, change it before you start. Unassigned tasks have no owner and go nowhere.

❌ If there is a discrepancy — wrong person assigned, task sitting with a dev while QA should own it — fix it immediately. Don't leave it.

✅ One clear owner at every stage. Dev owns In Progress. You own Testing & Validation.

Problem & Actual Results

REQUIRED

When filing a bug from the Backlog screen, the task description field becomes your bug report. Use the template — it auto-populates when you create from Backlog. Add detail. Do not leave it at the default.

Describe exactly what happened when running the test: what you did, what the app showed, and how it differs from expected. Attach a screenshot or screen recording whenever possible. A picture removes ambiguity instantly.

❌ "It's broken" or "doesn't work on mobile" tells the developer nothing they can act on.

✅ "When I tap [−] with qty at 1, the quantity decrements to 0 and no toast appears. Expected: qty stays at 1 and toast shows. See attached recording."

Steps to Reproduce

ALWAYS REQUIRED

Always fill this in — even if it is just two short steps. This is the single most useful thing you can give a developer. Without it, they cannot find the bug, cannot fix it, and cannot confirm the fix.

Number each step. Be exact. Include the starting state, the action, and the exact input if relevant.

❌ Leaving this blank because "it seems obvious" — it is never obvious to someone who wasn't there.

✅ Even a two-liner helps: 1. Log in as Buyer. 2. Add any item to cart, tap [−] when qty is 1.

Priority

IMPORTANT

Urgent — blocks release or breaks a critical user flow. High — significant impact but workaround exists. Normal — standard work. Low — cosmetic, won't block shipping.

Priority is Business. How much does this hurt the customer or the release? That is priority.
Severity is Software. How badly is the system broken technically? That is severity.
A broken font on a checkout page can be low severity (cosmetic) but high priority (customer-facing revenue flow).

Areas

IMPORTANT

Areas make tasks filterable and groupable across lists. Apply the area that matches the part of the product affected. For QA outcomes, also add a result area.

QA result areas: qa:pass   qa:fail   qa:not a bug   qa:not reproduceable

Product areas: mobile   site   cart   storefront   payments   wireframe

Due Date

OPTIONAL

Set only when there is a real deadline tied to a sprint boundary or release date. A wrong or fabricated due date is worse than no date — it creates false urgency and noise.

✅ Sprint 41 ends May 31 — if a task must ship in this sprint, set May 31.

Linked Tasks / Dependencies

OPTIONAL BUT VALUABLE

Link bug tasks back to the test case that found them. Link test cases to the feature task they cover. This creates a traceable chain: Story → Dev Task → Test Case → Bug.

✅ A linked bug tells the developer exactly which scenario failed without them having to hunt for context.

⚠️ THE RULE

A bug task with no Assignee, no Steps to Reproduce, and no Expected vs Actual will sit untouched forever. Dev can't fix what they can't reproduce. QA can't verify what they can't trace.

✅ DO

❌ DON'T

Write "it doesn't work" or "broken on mobile." That tells nobody anything. Don't assign priority "urgent" to everything — it means nothing if everything is urgent.
